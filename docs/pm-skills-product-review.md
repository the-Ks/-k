# PM Skills 产品评审

本文基于 PM 产品经理套件中的 PRD、用户故事、测试场景、优先级、指标看板、客户旅程、路线图和风险预演方法，对客服质检系统做一次产品化整理。

## 1. 产品目标

把淘宝和微信里的多来源聊天记录，变成可追溯、可质检、可复核、可管理决策的数据系统。

核心目标不是“展示聊天记录”，而是：

```text
多来源聊天数据
-> 身份归一
-> 会话链路
-> 质检评分
-> 客户画像
-> 管理动作
```

## 2. 北极星指标

| 指标 | 定义 | 为什么重要 |
| --- | --- | --- |
| 可质检闭环会话率 | 已完成身份匹配、质检评分、人工复核的会话数 / 总会话数 | 代表系统是否真正把聊天数据转成管理价值 |

本期 One Metric That Matters：

```text
身份匹配准确率
```

原因：如果淘宝和微信身份对不上，后面的评分、客户画像、BI 都会失真。

## 3. 用户故事

### 超级管理员创建账号

作为超级管理员，我想为新的客服或质检员提交账号申请，以便员工能按岗位进入系统。

验收标准：

- 可以填写姓名、登录账号、部门、角色、数据范围。
- 提交后进入待开通账号列表。
- 前端不保存数据库密码或云数据库连接信息。
- 后端接口预留写入云数据库的位置。

### 质检员复核身份匹配

作为质检员，我想看到系统推荐的淘宝和微信身份匹配结果，以便确认客户是否为同一人。

验收标准：

- 能看到淘宝账号、微信账号、匹配置信度。
- 能看到匹配依据来自哪条聊天消息。
- 可以确认、驳回、标记完成。
- 低置信度记录必须进入人工复核。

### 管理者查看 BI

作为管理者，我想看到客服质检趋势和异常指标，以便判断服务质量是否改善。

验收标准：

- 看板必须说明统计周期和指标口径。
- 平均分必须说明是客服综合质检分。
- 响应时长必须说明是否排除了自动回复和系统消息。
- 指标必须支持趋势比较和异常预警。

## 4. 优先级 Top 5

| 排名 | 事项 | 理由 | 优先级判断 |
| --- | --- | --- | --- |
| 1 | 云数据库接入 | 没有真实数据，系统只能演示 | 高影响 / 高确定性 |
| 2 | 身份归一闭环 | 淘宝和微信客户必须先对上 | 高影响 / 中等工作量 |
| 3 | 质检证据链 | 每个分数都要能追到消息证据 | 高影响 / 中等风险 |
| 4 | AI 质检接入 | 用于专业度、满意度、潜客意向判断 | 中高影响 / 需验证 |
| 5 | BI 预警 | 让管理层看到异常并采取动作 | 中影响 / 低工作量 |

## 5. 客户旅程

| 阶段 | 客户动作 | 系统机会 |
| --- | --- | --- |
| 淘宝咨询 | 询问价格、效果、售后 | 识别首次问题和客服首响时间 |
| 引导加微信 | 客户进群并发送淘宝ID | 抽取身份线索并匹配身份 |
| 微信群答疑 | 多个服务人员共同回复 | 识别负责人、专业度、无人承接 |
| 质检复核 | 质检员查看评分证据 | 修正误判并沉淀规则 |
| 管理决策 | 管理层看趋势和异常 | 驱动培训、排班和流程优化 |

## 6. 指标看板设计

| 指标 | 定义 | 数据源 | 展示 | 预警 |
| --- | --- | --- | --- | --- |
| 可质检闭环会话率 | 完成复核会话 / 总会话 | conversation + quality_score | 趋势线 | 低于 80% |
| 身份匹配准确率 | 人工确认正确数 / 抽检数 | person_account_map | 数字 + 趋势 | 低于 95% |
| 首次响应中位数 | 客户问题到人工回复中位时间 | standard_message | 分布图 | 高于 180 秒 |
| 平均质检分 | 已质检会话总分平均值 | quality_score | 趋势条 | 连续 3 天下滑 |
| 高意向客户数 | 高意向标签客户数 | customer_tag | 数字 + 趋势 | 异常下降 |

## 7. 结果导向路线图

### 第一阶段：让聊天记录能被系统可靠读入

交付：

- 云数据库接口
- 原始消息表
- 同步日志
- 字段标准化

### 第二阶段：让淘宝和微信里的同一个人能对上

交付：

- 身份线索抽取
- 账号映射表
- 人工复核
- 统一人员档案

### 第三阶段：让质检结果可解释、可复核

交付：

- 响应时间规则
- 流程合规规则
- 评分证据
- 复核状态

### 第四阶段：让 AI 辅助判断复杂语义

交付：

- AI Prompt
- AI 占位接口
- 模型配置
- 人工确认闭环

### 第五阶段：让管理层能根据数据做管理动作

交付：

- BI 看板
- 预警阈值
- 客服排行
- 客户意向趋势

## 8. 测试场景

### 新建账号申请

起始条件：

- 超级管理员已登录。

步骤：

1. 进入账号权限页。
2. 点击新建账号申请。
3. 填写姓名、登录账号、部门、角色、数据范围。
4. 提交申请。

预期结果：

- 弹窗关闭。
- 待开通账号列表出现新记录。
- 后端 `/api/accounts/request` 返回成功。

### 身份复核

起始条件：

- 存在低置信度身份匹配任务。

步骤：

1. 进入身份复核页。
2. 查看系统推荐对象和证据。
3. 点击确认或驳回。

预期结果：

- 任务状态更新。
- 匹配依据仍可追溯到消息。

### AI 质检占位

起始条件：

- 后端服务已启动。

步骤：

1. 调用 `/api/quality/ai-evaluate`。
2. 传入 `conversation_id`。

预期结果：

- 返回 `aiConnected: false`。
- 返回 prompt 文档路径。
- 不伪装成已经接入真实 AI。

## 9. 风险预演

### 真实风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 云数据库字段不稳定 | 数据同步失败、质检结果错误 | 先定义标准消息结构和字段映射 |
| 身份匹配错误 | 客服评分和客户画像失真 | 低置信度必须人工复核 |
| AI 判断不可解释 | 质检员不信结果 | AI 结果必须返回证据消息ID |

### 待验证风险

| 风险 | 验证方式 |
| --- | --- |
| AI 专业度判断是否可靠 | 抽取 100 条会话做人机对比 |
| BI 指标是否能驱动管理动作 | 让管理层每周看一次并记录行动 |
| 客服是否接受评分机制 | 先在小团队试运行 |

## 10. 下一步建议

先做一个真实数据闭环：

```text
1 个真实客户
-> 淘宝聊天
-> 微信群聊天
-> 身份匹配
-> 质检评分
-> 人工复核
-> BI 展示
```

这个闭环跑通，比继续增加页面更重要。
